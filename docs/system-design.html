01  Requirements

Functional

  • Accounts; private decisions that persist and sync across devices.
  • Each decision = options + criteria (weight, uncertainty) + per-participant scores.
  • Run the Monte-Carlo engine and store reproducible results (P(best), EVPI, dominance, tornado, EVPPI).
  • Teams: multiple stakeholders score independently; surface where they disagree.
  • Template gallery; generated decision-record reports; outcome journal for calibration.
  • Free / Pro / Team billing tiers.

Non-functional

  • Scale from day one: design for 100k+ users, no single-tenant assumptions.
  • Interactive sim feedback < 100 ms (client); authoritative sim < ~2 s (server).
  • Hard multi-tenant isolation — a user can never read another's decision.
  • Reproducibility: identical inputs ⇒ identical result (seeded Monte Carlo).
  • Real-time collaboration latency < 1 s for co-editing a team decision.
  • Reads cheap and cacheable; heavy compute isolated so it can't stall the API.

Constraints

Small team, cost-sensitive until revenue, Supabase as the platform (Postgres + Auth + RLS + Realtime + Edge Functions + Storage). Favor managed primitives over bespoke infrastructure; keep the verified TypeScript engine as the single source of truth for the math on both client and server.

The client talks to four Supabase surfaces: Auth for identity, PostgREST for all CRUD (every request filtered by Row Level Security, so the database itself is the authorization boundary), Realtime for live team collaboration, and Edge Functions for authoritative compute and report generation. There is no bespoke application server — Postgres + RLS is the backend, which is what keeps a small team able to operate at scale.

03  The compute split (the key decision)

The engine is identical TypeScript on both sides; where it runs depends on what the run is for.

Run typeWhereWhy
Live "what-if" while dragging a weight sliderClientInstant (<100 ms), free, fully private. No round-trip, no cost, works offline.
The decision's authoritative resultEdge fnReproducible & seeded, stored in simulations, citable in a report, identical for every viewer.
Group / consensus run across participantsEdge fnNeeds every participant's inputs — the server already has them under RLS; avoids trusting one client.
Report / decision-record generationEdge fnServer owns the canonical numbers and writes the PDF to Storage.

Reproducibility by construction

Both engines seed a mulberry32 PRNG from an FNV-1a hash of the decision inputs, so the same decision always yields the same P(best), intervals, and EVPPI — no flicker on refresh, and a stored result can be re-derived and audited. The server port was verified against the client engine: identical headline on the reference case (leader 52%, EVPI 2.72, top information-value factor = "Growth").

Why synchronous now, queued later

A standard run (6k trials) and even nested-Monte-Carlo EVPPI (120×120) complete well inside an Edge Function's budget, so simulate-decision computes inline and inserts a finished simulations row in one call. The escape hatch for very large runs is already modeled: a queued → running → done status column and the pgmq queue extension let heavy jobs move to a worker without changing the data contract.

04  Data model

The pivotal modeling choice is that weights and scores are per-participant, not per-decision. A solo decision is just the single-participant case; a team decision reuses the exact same tables. That one decision is what makes group decisions, disagreement-surfacing, and "decide as yourself then compare" fall out naturally instead of needing a separate system.

  • Scores are uncertainty, not points — criteria.uncertainty (low/med/high → ±1/±2/±3) drives the triangular bands the engine samples, so the uncertainty that powers EVPPI lives in the schema.
  • Results are immutable snapshots — a simulations row stores its seed, trials, engine_version, and a result JSONB blob, so a decision made last quarter can be reproduced and explained later.
  • Outcomes close the loop — decision_outcomes records what was chosen, a review_due date, and how it turned out, feeding the calibration/decision-journal features.
  • Billing is polymorphic — a subscription attaches to a user or a team (check num_nonnulls(user_id, team_id) = 1).

05  Access control & security

Authorization lives entirely in the database via Row Level Security — there is no app-server trust boundary to get wrong. Six SECURITY DEFINER helper functions answer the access questions, and policies call them:

-- the heart of multi-tenant isolation
can_read_decision(d) = owner OR team member OR named participant
can_edit_decision(d) = owner OR team admin OR participant{owner|facilitator}
owns_participant(p)  = the participant row belongs to auth.uid()
  • No recursion — helpers are SECURITY DEFINER with search_path = '', so a policy on decisions can query decisions without re-triggering its own policy (the classic Supabase RLS trap, avoided).
  • Separation of duties — facilitators edit structure (options, criteria, roster); contributors may only write their own input_weights/input_scores (enforced by owns_participant), but can read everyone's to see the group picture.
  • Billing is read-only to clients — subscriptions has a SELECT policy but no write policy; only the Stripe webhook (service role, which bypasses RLS) mutates it.
  • Hardened — every table has RLS enabled; FK covering indexes added; auth.uid() wrapped in scalar sub-selects to evaluate once per query, not per row; helper EXECUTE revoked from anon.

Known, accepted advisor items

The six RLS helpers remain EXECUTE-able by authenticated — this is required for RLS to evaluate them, and they leak nothing (each returns only the caller's own boolean access). The textbook next step is relocating them to a non-API-exposed private schema. Separately, a pre-existing public.rls_auto_enable() function (not part of this build) is flagged and should be reviewed.

06  API contracts

Data plane — PostgREST (auto-generated, RLS-filtered)

CRUD needs no hand-written endpoints. The client uses supabase-js with generated types; the database decides what's visible.

  .eq('id', decisionId).single()

Realtime — collaboration channels

input_scores, input_weights, decision_participants, and simulations are published to Realtime. A team decision subscribes to row changes scoped to its decision_id, so a teammate's score appears live and a finished simulation pushes to everyone.

07  Deep dive — team decisions & disagreement

Because inputs are per-participant, a team decision is a fan-in/fan-out over the same tables:

  1. Set up — a facilitator defines options & criteria and adds participants (existing users or email invites → decision_participants).
  2. Independent input — each participant writes only their own input_weights/input_scores; Realtime streams presence and progress without exposing who-said-what prematurely.
  3. Aggregate — simulate-decision with aggregation = mean|median reduces participants per (option, criterion) and per weight, then runs the standard engine.
  4. Surface disagreement — the spread of weights per criterion and scores per cell is the signal: high variance on a high-weight criterion is exactly where the team should talk. (This dovetails with EVPPI: "you disagree most on the thing that matters most — resolve it.")

Why server-side aggregation

No single client can be trusted to hold everyone's inputs, and RLS lets contributors read peers' inputs but not edit them. The Edge Function runs with the caller's JWT, so it can only aggregate a decision the caller is actually part of — authorization and computation share one boundary.

08  Deep dive — value of information at scale

EVPPI is the product's signature output and its most expensive computation: a nested Monte Carlo (outer × inner per criterion). Cost ≈ M × K × options × criteria. The plan keeps it cheap where it can and isolates it where it can't:

  • Tiered depth — the function caps nested work and skips EVPPI when options × criteria is large, so a pathological decision can't hang a request.
  • Cache by input hash — results are deterministic, so the seed/hash doubles as a cache key; an unchanged decision never recomputes.
  • Offload path — the simulations status machine + pgmq let a 50k-trial or many-participant run enqueue and complete on a worker, with Realtime notifying the client when status = done.

09  Scale & reliability

Where the load is

  • Reads dominate (open/list decisions) — cheap, indexed, RLS-filtered.
  • Writes are small and bounded (one user's scores).
  • CPU concentrates in simulation — deliberately pushed to stateless Edge Functions that scale horizontally and independently of the API.

Scaling levers

  • Supavisor connection pooling for serverless fan-out.
  • Read replicas when read volume climbs.
  • pgmq worker pool for heavy/most-expensive runs.
  • Partition simulations by month (pg_partman) once it's the largest table.

11  What was actually built (live in the project)

  • 8 migrations → 14 tables, 9 enums, 6 RLS helper functions, new-user & team-owner triggers, updated_at triggers.
  • RLS on all 14 tables with read/write policies; FK covering indexes; per-query auth.uid(); anon execute revoked.
  • Edge Function simulate-decision (ACTIVE, JWT-verified) — full engine: simulate, dominance, tornado, nested-MC EVPPI; DB-assembly and stateless-payload modes.
  • 4 official templates seeded (job offer, hiring, vendor, where-to-live).
  • Realtime enabled on the four collaboration tables; TypeScript types generated for the client.
  • Verified — server engine reproduces the client headline exactly (leader 52%, EVPI 2.72, EVPPI top = Growth); security & performance advisors run and remediated.

12  What I'd revisit as it grows

  • Move RLS helpers to a private schema to clear the residual advisor warnings entirely.
  • Promote the engine to a shared package imported by both client and function, so the two ports can never drift.
  • Materialize team aggregates (a per-decision rollup) once group decisions are common, to avoid re-reducing inputs on every run.
  • Link-sharing & anonymous input via signed tokens through an Edge Function (kept out of RLS deliberately for now).
  • Partition + archive simulations, and add an EVPPI result cache table keyed by input hash.
  • Stripe webhook + report-render functions are designed (status machine, Storage, polymorphic subscriptions) but not yet implemented — the next build.
